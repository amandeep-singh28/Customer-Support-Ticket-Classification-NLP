(i)Observation 1: Presence of Noisy Words in the Dataset

Problem
While exploring the Document column, it was observed that many tickets contain common conversational or email-specific words that may not contribute meaningful information for ticket classification.

Examples include:

dear
thanks
thank you
hello
hi
regards
kind regards
good morning
please
lead

These words are commonly used in email communication and appear across multiple ticket categories. Since they do not describe the actual issue, they are considered potential noisy words.

Impact

If retained, these words may:

Increase the vocabulary size unnecessarily.
Increase the dimensionality of the feature space.
Add noise to the model.
Reduce model interpretability.

However, when using TF-IDF, words that appear in a large number of documents automatically receive a very low IDF score, reducing their influence on the final feature vectors.

Proposed Solution

Instead of manually removing every suspected noisy word, the project will follow an experimental approach:

(i)Train a baseline model using the original text with TF-IDF.
(ii)Evaluate the model's performance.
(iii)Analyze the most influential features learned by the model.
(iv)If common conversational words are found to negatively affect performance, create a custom stopword list and retrain the model.
(v)Compare both models and keep the preprocessing strategy that yields better performance.

(ii)Observation 2: TF-IDF Handles Common Words Effectively
Problem

Initially, it was assumed that the presence of common conversational words such as:

hello
thanks
dear
regards
please
good morning

would significantly reduce the model's performance.

Observation

After training the baseline model using TF-IDF + Logistic Regression without removing these words, the model achieved:

Accuracy : 86%

This indicates that the presence of these common words did not severely affect the classification performance.

Reason

TF-IDF automatically assigns very low weights to words that appear in a large number of documents by reducing their IDF score.

As a result, informative words such as:

vpn
oracle
printer
mailbox
password
wireless

receive much higher importance than common conversational words.

Conclusion

At this stage, removing custom stopwords is not immediately necessary.

Instead, any preprocessing should be performed only if it provides measurable improvements over the baseline model.

Looking at the Classification Report:

fewer training samples

(v)Observation 5

A preprocessing pipeline was created to remove conversational words, punctuation, numbers and stopwords while preserving domain-specific technical terms.

This reduces textual noise and helps the model focus on informative words during feature extraction.

(vi)Observation 6

A text preprocessing pipeline was implemented by converting text to lowercase, removing numbers, punctuation, English stopwords, and custom stopwords.

The preprocessed text was used to retrain the Logistic Regression model.

Result:
The overall classification performance remained almost unchanged, with slight improvements in a few classes and minor degradation in others.

Conclusion:
Since TF-IDF already assigns very low weights to highly frequent words, removing these words did not significantly improve the model. Therefore, the baseline text representation is retained for further model comparison.

(vii)Observation 7

Two preprocessing experiments were conducted by gradually increasing the custom stopword list.

Result:

No significant improvement in model performance was observed.

Conclusion:

The TF-IDF representation was already reducing the influence of highly frequent conversational words. Additional manual stopword removal slightly degraded performance for several classes.

Decision:

The original Document column will be used for subsequent experiments.

(viii)Observation 8

Instead of manually training each classifier separately, a dictionary of machine learning models is created.

This allows every model to be trained and evaluated using the same preprocessing pipeline, ensuring a fair comparison while reducing code duplication.

(ix)Observation 9

Each model is evaluated using four metrics:
Accuracy, Precision, Recall, and F1 Score.

Since this is a multi-class classification problem with an imbalanced class distribution, weighted averaging is used to compute Precision, Recall, and F1 Score. This ensures that each class contributes to the final metric proportionally to its frequency in the dataset.

(x)Observation 10
Observation 11: The evaluation metrics of each model are stored in a list of dictionaries. After all models have been trained and evaluated, this list is converted into a Pandas DataFrame, making it easy to compare the performance of different machine learning algorithms side by side.

(xi)Different machine learning algorithms perform differently on the same TF-IDF features.

Logistic Regression achieved the highest overall Accuracy and F1 Score, followed very closely by Linear SVM.

Random Forest performed slightly worse because tree-based models generally do not handle high-dimensional sparse TF-IDF features as effectively as linear models.

Multinomial Naive Bayes produced the lowest performance because it assumes that all words are conditionally independent, which is often unrealistic for real customer support tickets.
